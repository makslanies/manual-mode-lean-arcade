# РУЧНОЙ РЕЖИМ — Технический документ игры
### Аркада для тех, кто управляет производством · Прототип актов 1–3

**Версия:** 1.0 · **Дата:** 19.07.2026 · **Основание:** act1-prototype.html (одностраничное HTML5/Canvas-приложение, ~68 КБ, без внешних зависимостей)

## 1. Общие сведения

### 1.1. Назначение продукта
«Ручной режим» — короткая (4–5 минут) браузерная аркада-симулятор директора завода. Игрок в реальном времени тушит производственные «пожары», затем получает датчики (наблюдаемость), затем — автоматические правила (логика), и на собственном опыте проживает лестницу зрелости управления производством:

**Хаос вручную → Наблюдаемость (датчики) → Автоматизация (правила ЕСЛИ→ТО)**

Игра является демонстрационно-обучающим инструментом («реальная платформа делает то же с настоящим заводом» — финальный экран), т.е. одновременно маркетинговым лендингом промышленной IIoT/MES-платформы и тренажёром для обучения принципам бережливого производства.

### 1.2. Целевая аудитория
- Директора и руководители производств, главные инженеры, начальники цехов.
- Участники тренингов по бережливому производству / TPS / цифровизации.
- Посетители выставочного стенда (предусмотрен сервисный хук `window.__RR` для демонстраций).

### 1.3. Ключевой дизайн-принцип
Проигрыш в первом акте — запланированный педагогический результат. Титульный экран прямо сообщает: «не успеешь — и это нормально». Игрок должен физически ощутить нехватку внимания и рук, чтобы ценность датчиков и автоматики была пережита, а не рассказана.

### 1.4. Технологический стек
| Компонент | Решение |
|---|---|
| Платформа | Один HTML-файл, HTML5 Canvas 2D, Vanilla JS (IIFE, strict mode) |
| Графика | Процедурная изометрия (без спрайтов и ассетов) |
| Звук | Web Audio API, процедурный синтез (осцилляторы) |
| UI-оверлеи | DOM поверх canvas (title/supply/rules/end), CSS-переменные |
| Ввод | Pointer Events + клавиатура (WASD/стрелки/ФЦВЫ, Space/Enter) |
| Зависимости | Отсутствуют; работает офлайн |

## 2. Логические уровни игры (машина состояний)

Игра построена как конечный автомат из 7 состояний (`mode`):

```
title ──► play (Акт 1) ──► supply (Пауза снабжения) ──► play2 (Акт 2)
              130 c              15 c                        55 c
                                                              │
        end (Итоги) ◄── play3 (Акт 3) ◄── rules (Инженерная пауза)
                             60 c                20 c
          └─── «ЕЩЁ СМЕНА» ──► resetGame() ──► play
```

| Состояние | Название в UI | Длительность | Тип | Что происходит |
|---|---|---|---|---|
| `title` | Титульный экран | ∞ | Оверлей | Завязка, кнопка «НАЧАТЬ СМЕНУ» |
| `play` | СМЕНА · ВРУЧНУЮ | `ACT1_LEN = 130 с` | Геймплей | Акт 1: ручное управление, зоны открываются по таймеру, туман войны |
| `supply` | Пауза снабжения | `SUPPLY_LEN = 15 с` | Оверлей | Выбор 3 датчиков из 5 (авто-монтаж по истечении таймера) |
| `play2` | СМЕНА · С ДАТЧИКАМИ | `ACT2_LEN = 55 с` | Геймплей | Акт 2: зоны с датчиками видны постоянно, ранние тревоги |
| `rules` | Инженерная пауза | `RULES_LEN = 20 с` | Оверлей | Подключение до 3 правил ЕСЛИ→ТО (автозапуск по таймеру) |
| `play3` | СМЕНА · С ЛОГИКОЙ | `ACT3_LEN = 60 с` | Геймплей | Акт 3: правила срабатывают сами, игрок ловит исключения |
| `end` | Конец смены | ∞ | Оверлей | Сравнение потерь по актам, кнопка «ЕЩЁ СМЕНА» |

Переходы жёстко по фазовому таймеру `phaseT`; паузы имеют собственный обратный отсчёт (`supplyLeft`, `rulesLeft`) и автозавершение, чтобы стендовый режим никогда не «зависал» на выборе.

### 2.1. Акт 1 — «Вручную» (130 с): обучение через перегрузку
**Цель уровня (игровая):** удержать завод, кликая по зонам.
**Цель уровня (педагогическая):** довести игрока до когнитивной перегрузки и показать цену слепоты.

Драматургия акта — постепенное открытие зон по расписанию `z.unlock`, каждая новая зона добавляет канал внимания:

| Время | Открывается зона | Новая механика |
|---|---|---|
| 0 с | СТАНОК «АТЛАНТ» | Перегрев → авария |
| 15 с | БУНКЕР СЫРЬЯ | Расход сырья → остановка линии |
| 30 с | ПОГРУЗОЧНЫЙ ДОК | Дедлайны рейсов, логистика |
| 50 с | ПРЕСС «ЦИКЛОП» | Износ; зависит от станка (цепочка) |
| 68 с | ПОСТ СБОРКИ | Усталость рабочего |
| 88 с | СКЛАД-ХОЛОД | Порча партии, потеря груза |
| 105 с | ЭНЕРГОЩИТ | Каскадный отказ всего завода |

Сложность растёт линейно: `difficulty = lerp(1 → 2.2)` по ходу акта. Сюжетные подсказки (2 с, 34 с, 58 с, 110 с) ведут игрока, последняя («Не успеваешь? Это нормально») легитимизирует поражение.

**Туман войны:** без датчика внутреннее состояние зоны скрыто (силуэт + «?»). Заглянуть внутрь можно только подойдя/кликнув — зона видна 3 секунды (`lookUntil = gameT + 3`). Это симуляция обхода цеха директором.

### 2.2. Пауза снабжения (15 с): урок приоритизации
Бюджета хватает на **3 датчика из 5** (`SENSOR_BUDGET = 3`): термодатчик станка, датчик уровня бункера, GPS-телематика рейсов, термодатчик склада, счётчик энергии. Пресс и пост сборки датчиков не имеют — принципиально: наблюдаемость не бывает тотальной, выбирать нужно «где болело сильнее». Если игрок не выбрал — автомонтаж берёт первые три по фиксированному приоритету (станок, энергощит, док, склад, бункер).

При входе в Акт 2 состояние «обнуляется» для честного сравнения: все зоны включены, `value ≤ 0.4`, отказы сняты, `powerOut = false`, дедлайны рейсов сдвинуты.

### 2.3. Акт 2 — «С датчиками» (55 с): наблюдаемость
Зоны с датчиками видны постоянно (`zoneVisible = z.sensor || lookUntil`). Появляется механика ранней тревоги: при `value ≥ 0.7` (для дока — менее 8 с до дедлайна) зона получает жёлтую пульсацию, звук `sAlarm`, флоатер «ТРЕВОГА». Все зоны активны сразу, сложность фиксирована 2.2 — выше, чем в начале Акта 1, но игрок теперь предупреждён заранее. Ключевой опыт: те же руки, тот же темп, но реагировать можно до аварии → рост «ПРЕДОТВРАЩЕНО» и комбо-серий.

### 2.4. Инженерная пауза (20 с): проектирование логики
Игрок соединяет «кабелем» строки ЕСЛИ→ТО, доступны только правила для зон **с датчиками** (прямое следствие выбора в паузе снабжения: «Без датчиков соединять нечего — логика строится на данных»). Лимит `RULE_BUDGET = 3`. При бездействии — автовыбор первых доступных правил.

### 2.5. Акт 3 — «С логикой» (60 с): автоматизация + человек на исключениях
Правила срабатывают сами (кулдаун 1.4 с на правило), по кабелю от «ЩИТА ЛОГИКИ» к зоне летит импульс, всплывает «АВТО», растёт счётчик `autoActs`. Сложность максимальна (2.4), но рутина закрыта автоматикой — игрок работает только с исключениями: зоны без датчиков (пресс, сборка), сброс энергощита, нехватка груза. Подсказка акта: «Логика работает — лови исключения!».

### 2.6. Итоги: доказательство деньгами
Финальный экран сравнивает **процент потерь** по актам: `pct = lost / (lost + earned)`. Заголовок динамический: «Логика окупилась» (если p3 < p1), иначе «Хаос силён — попробуй ещё». Выводятся: потери по трём актам (₽ и %), рейсы вовремя/всего, «Предотвращено · автоматикой», касса смены. Явный мостик к продукту: «Так работает лестница: наблюдаемость → автоматизация».

## 3. Игровой мир и объекты

### 3.1. Изометрическая сцена
- Сетка `GRID = 18×18`, тайл `TW=68, TH=34` (2:1 diamond).
- Преобразования: `isoPt(gx,gy) = ((gx−gy)·TW/2, (gx+gy)·TH/2)`; обратное `screenToGrid` для кликов по полу.
- Камера автоподгоняется под окно: `camScale = clamp(min(W/1400, (H−70)/900), 0.45, 1.1)`, центр сетки → точка (W/2, 0.52·H). Ресайз пересчитывает DPR (cap 2) и камеру.
- Дорога: колонна `gx 14–16` + подъезд к доку `gy 8–9`; ворота вывоза у (15.5, 1.6); разметка пунктиром.

### 3.2. Зоны (полная спецификация)

| id | Название | Позиция (gx, gy) | Открытие (Акт 1) | Шкала | Датчик | Скорость роста value (×difficulty) |
|---|---|---|---|---|---|---|
| `machine` | СТАНОК «АТЛАНТ» | 3.0, 3.0 | 0 с | T° СТАНКА | ТЕРМОДАТЧИК СТАНКА | +0.026/с (когда производит) |
| `hopper` | БУНКЕР СЫРЬЯ | 8.0, 2.5 | 15 с | СЫРЬЁ (инверт.) | ДАТЧИК УРОВНЯ БУНКЕРА | +0.020/с (когда станок работает) |
| `dock` | ПОГРУЗОЧНЫЙ ДОК | 12.0, 8.0 | 30 с | ГРАФИК (дедлайн) | GPS-ТЕЛЕМАТИКА РЕЙСОВ | — (таймеры рейсов) |
| `press` | ПРЕСС «ЦИКЛОП» | 3.0, 7.5 | 50 с | ИЗНОС | — (нет датчика) | +0.018/с (когда есть детали) |
| `worker` | ПОСТ СБОРКИ | 8.0, 7.0 | 68 с | СИЛЫ | — (нет датчика) | +0.024/с (×0.4 при блэкауте) |
| `fridge` | СКЛАД-ХОЛОД | 3.0, 12.0 | 88 с | T° СКЛАДА | ТЕРМОДАТЧИК СКЛАДА | +0.020/с ×(0.6..1.4 шум) ×1.7 при блэкауте |
| `power` | ЭНЕРГОЩИТ | 8.0, 12.0 | 105 с | НАГРУЗКА | СЧЁТЧИК ЭНЕРГИИ | +(0.008 + 0.007·N производящих)/с |

Служебный объект: **ЩИТ ЛОГИКИ** (`HUB` в 11.0, 12.2) — появляется в Акте 3, к нему тянутся кабели активных правил.

Состояние зоны: `{on, unlockAt, value 0..1, failed, lookUntil, sensor, warned, spawnT}`.

### 3.3. Производственная цепочка (взаимозависимости)
```
БУНКЕР ─сырьё─► СТАНОК ─детали─► ПРЕСС ─► ПОСТ СБОРКИ ─продукция─► СКЛАД/ДОК ─► РЕЙСЫ
                   ▲                                                    
ЭНЕРГОЩИТ ─питание─┴── (блэкаут останавливает станок, пресс, сборку; греет склад ×1.7)
```
Формализация в коде:
- `machineFlow()` = станок включён, не сломан, есть питание, бункер не пуст;
- `pressFlow()` = пресс жив + `machineFlow`;
- `chainFlow()` = пресс есть ? `pressFlow` : `machineFlow`;
- выработка готовой продукции: `stock += (1.0·machineFlow + 1.0·pressFlow + 1.2·workerFlow)·dt`, cap 99.

Педагогический смысл: игрок видит «НЕТ ДЕТАЛЕЙ» на прессе и учится искать корневую причину выше по потоку (флоатер прямо подсказывает: «ЖДЁТ СТАНОК — ЧИНИ ЕГО!»).

### 3.4. Отказы (failZone) — таблица последствий

| Зона | Условие | Событие | Штраф | Побочные эффекты |
|---|---|---|---|---|
| Станок | value ≥ 1 | АВАРИЯ | −220 ₽ | accident++, дым, тряска экрана, hitstop |
| Пресс | value ≥ 1 | КЛИН ПРЕССА | −180 ₽ | accident++, дым |
| Бункер | value ≥ 1 | БУНКЕР ПУСТ | — (косвенно: линия стоит) | станок/пресс/сборка не производят |
| Сборщик | value ≥ 1 | УСНУЛ | — (косвенно) | «zzz», выпадает из выработки |
| Склад | value ≥ 1 | ПАРТИЯ ИСПОРЧЕНА | −260 ₽ и **груз ×0.5** | accident++, value откат до 0.8 |
| Энергощит | value ≥ 1 | ВЫБИЛО АВТОМАТ | −180 ₽ | accident++, `powerOut=true`: весь завод стоит, красная вуаль экрана, склад греется ×1.7 |
| Док | дедлайн истёк, машина не ушла | РЕЙС СОРВАН | −160 ₽ | accident++, tripsLate++, график сдвигается на +20 с |

Дополнительный тлеющий урон склада: при `value ≥ 0.85` — −8 ₽/с и −1.2 ед. груза/с («ТЕПЛЕЕТ»).

### 3.5. Действия директора (actionFor) — таблица

Директор (аватар) идёт к зоне (скорость 3.6 клетки/с) и выполняет действие с прогресс-кольцом:

| Зона | Условие | Действие (label) | Длительность | Стоимость | Превентивное при |
|---|---|---|---|---|---|
| Станок | failed | ЧИНЮ… | 3.2 с | 80 ₽ | — |
| Станок | value ≥ 0.25 | ОБДУВ… | 1.0 с | — | value ≥ 0.7 |
| Пресс | failed | ЧИНЮ… | 2.6 с | 60 ₽ | — |
| Пресс | value ≥ 0.3 | СМАЗЫВАЮ… | 1.1 с | — | value ≥ 0.7 |
| Бункер | value ≥ 0.6 | ЗАКАЗЫВАЮ… | 1.2–1.5 с | 40 ₽ | value ≥ 0.7 |
| Сборщик | failed | БУЖУ… | 1.0 с | — | — |
| Сборщик | value ≥ 0.6 | КОФЕ… | 0.8 с | — | value ≥ 0.7 |
| Склад | value ≥ 0.3 | ОХЛАЖДАЮ… | 1.5 с | — | value ≥ 0.7 и не failed |
| Энергощит | powerOut | СБРАСЫВАЮ… | 2.0 с | — | — |
| Энергощит | value ≥ 0.6 | РАЗВОЖУ НАГРУЗКУ… | 1.2 с | — | value ≥ 0.7 |
| Док | есть готовая машина и груз ≥ 25 | ОТПРАВЛЯЮ… | 0.8 с | — | до дедлайна < 6 с |

Важные правила:
- Действия на станке/прессе недоступны при блэкауте (сначала сбрось автомат).
- **Превентивность не засчитывается вслепую:** если зона не была видна в момент прихода (`!wasVisible`), флаг `preventive` снимается — награда за предотвращение требует наблюдаемости. Это ядро обучающей механики.
- Клик по грузовику эквивалентен клику по доку (щедрый хитбокс).

### 3.8. Правила ЕСЛИ→ТО (Акт 3)

| id | Зона | ЕСЛИ | ТО | Эффект run() | Стоимость |
|---|---|---|---|---|---|
| `r_cool` | machine | T° станка ≥ 80% | включить обдув | value → 0.12 | — |
| `r_order` | hopper | сырьё ≤ 20% | автозаказ | value → 0, −40 ₽ | 40 ₽ |
| `r_chill` | fridge | T° склада ≥ 70% | усилить холод | value → 0.15, снять failed | — |
| `r_grid` | power | нагрузка ≥ 80% | развести нагрузку | value → 0.25 | — |
| `r_go` | dock | рейс горит (< 5 с) | автодиспетчер | dispatchTruck(auto) | — |

Механика исполнения: каждый активный rule проверяется в цикле, кулдаун `cd = 1.4 c` после срабатывания, `autoActs++`, визуал (импульс по кабелю, флоатер «АВТО», звук `sAuto`), 25% шанс реплики «Обработал сам, директор».

Дизайн-ограничение: правила покрывают только 5 зон с датчиками; пресс, сборка и сброс автомата остаются человеку → в Акте 3 игрок не безработный, а «работает по исключениям».

## 4. Практическое обучение (педагогический слой)

### 4.1. Карта соответствия «механика ↔ концепция бережливого производства»

Игра — интерактивная иллюстрация ключевых идей TPS/Lean (по материалам проекта: Ohno «Toyota Production System», Liker «The Toyota Way Fieldbook», Womack/Jones «Lean Thinking»):

| Игровая механика | Концепция | Комментарий для тренера |
|---|---|---|
| Туман войны, «заглянуть на 3 секунды» | Генти генбуцу («иди и смотри») и его пределы масштаба | Обход цеха работает, пока зон три; на семи — директор физически не успевает |
| Датчики + постоянные шкалы | Визуальный контроль / мieruka | Состояние процесса видно любому с одного взгляда, без обхода |
| Жёлтая тревога при value ≥ 0.7 → красная авария при 1.0 | Андон (зелёный/жёлтый/красный) | Тревога до отказа = окно реакции; в игре ≈ окно 5–30 c из практики Toyota |
| Флаг «превентивно» засчитывается только для видимой зоны | Признание проблем требует стандарта и сигнала | Нельзя предотвратить то, чего не видишь; награда привязана к наблюдаемости |
| Правила ЕСЛИ→ТО, срабатывающие без директора | Дзидока / автономизация («автоматизация с человеческим интеллектом») | Станок сам «принимает решение» (обдув при перегреве) — перенос интеллекта в машину; аналог автоостанова станка Тоёды при обрыве нити |
| Автозаказ сырья при остатке ≤ 20% | Канбан / вытягивающее пополнение по точке заказа | Пополнение по факту потребления, а не по расписанию |
| Директор в Акте 3 занимается только исключениями | Разделение труда человек/машина, «человек не сторож станка» | Освобождение человека от надзора за оборудованием для работы с проблемами |
| Цепочка бункер→станок→пресс→сборка, «НЕТ ДЕТАЛЕЙ» | Поток и поиск корневой причины (5 почему) | Симптом на прессе, причина на станке; подсказки учат смотреть выше по потоку |
| Срыв рейса, штраф и сдвиг графика | Такт/график поставок, цена нестабильности для клиента | «Клиент в ярости» — потери за пределами цеха |
| Блэкаут останавливает всё и греет склад | Системные взаимосвязи, каскадные отказы | Одна инфраструктурная точка отказа бьёт по всем переделам |
| stat.lost (упущенная выгода простоя) | Муда ожидания и недопроизводства | Потери считаются не только штрафами, но и недополученным доходом |
| Итоговое сравнение % потерь трёх актов | Измеримость улучшений, PDCA | Одна и та же смена в трёх режимах управления — контролируемый эксперимент |

### 4.2. Формируемые навыки (learning outcomes)
После сессии участник способен:
1. Объяснить разницу между реактивным управлением, наблюдаемостью и автоматизацией — на собственном опыте потерь в ₽ и %.
2. Приоритизировать точки установки датчиков при ограниченном бюджете (3 из 5) исходя из тяжести последствий отказа, а не «где привычнее».
3. Сформулировать управленческое правило в форме ЕСЛИ (порог по данным) → ТО (действие) и понять, что автоматика строится только на измеряемом.
4. Опознать роль человека в автоматизированном контуре: обработка исключений, а не дублирование автоматики.
5. Проследить симптом до корневой причины по производственной цепочке.

### 4.3. Сценарий использования на тренинге (30–40 минут)
1. **Брифинг (3 мин).** «Вы — директор. Задача — прибыль за смену». Правила не раскрывать глубже титульного экрана.
2. **Игровая сессия (5 мин).** Индивидуально или парами (один кликает, второй — «диспетчер», подсказывает голосом: заготовка под разговор о разделении наблюдения и действия).
3. **Съём результатов (2 мин).** Каждый фиксирует p1/p2/p3, аварии, рейсы, кассу (экран итогов).
4. **Дебриф (15 мин)** — вопросы фасилитатора:
   - Что вы чувствовали на 100-й секунде Акта 1? Где были ваши глаза, когда сломался склад?
   - Какие 3 датчика выбрали и почему? Кто пожалел о выборе? (мостик к анализу критичности отказов)
   - Что изменилось в Акте 2 при той же скорости ваших рук? (наблюдаемость ≠ автоматизация, но уже даёт эффект)
   - Что осталось на вас в Акте 3? Это похоже на вашу роль после цифровизации участка?
   - Где на вашем реальном производстве «зоны без датчиков»? Какое одно правило ЕСЛИ→ТО вы бы внедрили в понедельник?
5. **Перенос (10 мин).** Каждый участник формулирует для своего производства: 1 «слепую зону», 1 датчик, 1 правило. Связка с материалом курса: андон, дзидока, визуальный менеджмент, точка заказа.
6. **Повторная смена (опционально).** «ЕЩЁ СМЕНА» с другим набором датчиков — эксперимент со стратегией.

### 4.4. Встроенное обучение внутри игры (in-game teaching)
- **Прогрессия сюжетных подсказок** (`hintOnce`, показ один раз): «Кликни по станку», «Док открыт — отправляй рейсы», «Подойди к зоне — взгляни внутрь», «Не успеваешь? Это нормально», «Жёлтая тревога? Успей до аварии!», «Логика работает — лови исключения!».
- **Обучение через последствие**: hintOnce при первом событии каждого типа («Дым — уже поздно. Реагируй раньше!», «Линия встала — следи за бункером!», «Весь завод встал — сбрось автомат!», «График сорван — следи за доком!»).
- **Диагностика на месте**: подход к проблемной зоне без доступного действия выдаёт причинную подсказку («ЖДЁТ СТАНОК — ЧИНИ ЕГО!», «ЛИНИЯ ВЫШЕ ПО ЦЕПОЧКЕ СТОИТ», «НЕТ ГРУЗА — НУЖНА ЦЕПОЧКА»).
- **Рация (реплики персонажей)**: мастер, диспетчер, снабженец, энергетик, кладовщик озвучивают события; в простое — атмосферные idle-реплики каждые 13 с. Очередь ≤ 5, показ 3.2 с.
- **Тексты пауз** формулируют принципы явно: «Датчик не работает за тебя — он снимает туман и предупреждает заранее», «Правило срабатывает само, без директора», «Логика строится на данных».

### 5.5. Панель «ДИСПЕТЧЕРСКАЯ» (мини-дашборд, лево под HUD)
Список открытых зон, ×19px строка:
- зона с датчиком → мини-прогрессбар 64×8 (цвет по `gaugeColor`: <0.5 green, <0.75 amber, иначе red; red при активной проблеме) + «!» справа; в Акте 3 подпись «АВТО» у зон с активным правилом (вспышка 0.6 с после срабатывания);
- зона без датчика → серый текст «НЕТ ДАННЫХ»;
- если датчиков нет вообще — мигающая красная надпись «СЛЕПАЯ ЗОНА» (Акт 1: постоянная демонстрация проблемы);
- внизу: ГРУЗ n/25 (amber при нехватке) и РЕЙСЫ onTime/total.

Дашборд — это «экран платформы» внутри игры: игрок видит, как выглядела бы диспетчерская с телеметрией, и как она пуста без датчиков.

### 5.6. Внутримировой UI (над зонами)
- **Шкала зоны** (drawGauge): плашка с подписью (T° СТАНКА / СЫРЬЁ / …), бар 66×9; для бункера и дока показывается «остаток» (1−v). Под баром бейдж «СЕНСОР» (cyan), если датчик установлен.
- **Туман**: невидимая зона рисуется силуэтом (заливка `PAL.fog`), плавающий «?» и пылинки; подсветка пола серая вместо синей.
- **Проблемный бейдж**: красная пилюля «! ТЕКСТ» над зоной (АВАРИЯ / ПУСТО / СПИТ / ТЕПЛЕЕТ / ВЫБИЛО АВТОМАТ / НЕТ ПИТАНИЯ / НЕТ СЫРЬЯ / НЕТ ДЕТАЛЕЙ / НЕТ ПОТОКА / НЕТ ГРУЗА / РЕЙС ГОРИТ), покачивание по синусу. Для склада без датчика бейдж скрыт (порча «молчит» — усиливает урок).
- **Пульсация основания**: красная (проблема) или жёлтая (ранняя тревога по датчику) diamond-подсветка с пульсом.
- **Анимации зон**: жар станка (точка от жёлтого к красному по value), уровень сырья в бункере (cyan-столбик), ход пресса (синус при работе), порча коробок склада (зелёный→бурый градиент по value), молния энергощита (gold / red при блэкауте), стопки груза на доке (по 1 коробке на каждые 25 ед.), «ГРУЗ: n/25».
- **Грузовики**: кузов+кабина изобоксами, тень-эллипс; в статусе ready — маячок (green/amber/red по остатку времени) и, при видимости дока, таймер «N с» на плашке.
- **Директор**: тело/голова/золотая каска, боб-анимация шага, тень, пыль из-под ног; при работе — прогресс-кольцо (cyan-дуга) + label действия, искры.
- **Щит логики (Акт 3)**: здание с cyan-экраном, пульсирующее основание, надпись «ЩИТ ЛОГИКИ»; кабели — пунктирные квадратичные кривые (бег пунктира `lineDashOffset = −gameT·30`), импульсы срабатываний — cyan-шар с ореолом, летящий щит→зона.

## 6. Техническая архитектура

### 6.1. Главный цикл
- `requestAnimationFrame(frame)` → `tick(now)`; `dt = clamp(now−last, 0, 0.1)` — защита от скачков после сворачивания.
- Страховочный `setInterval` 250 мс: если rAF заморожен > 300 мс (фоновая вкладка), tick вызывается принудительно — таймеры пауз не «встают».
- Порядок кадра: `keyMove → update → render`. Оверлейные режимы обновляют только свои таймеры и дорисовывают ≤ 4 кадров.
- Hitstop реализован масштабированием dt (визуальный рендер идёт на реальном dtReal).

### 6.2. Порядок отрисовки (painter's algorithm)
`fillBG → shake-translate → drawFloor → drawLogic (кабели/щит) → зоны в порядке ZONES_ORDERED (сорт по gx+gy) с врезкой директора по глубине → drawTrucks → красная вуаль блэкаута → drawParticles → drawHUD → drawDashboard → drawQuip`.

### 6.3. Состояние (полный перечень глобалов)
`mode, gameT, phaseT, money, comboMult/N/Until, accidents, prevented, autoActs, stock, tripsOnTime/Late, powerOut, stat{1,2,3}, shake, hitstop, supplyLeft, rulesLeft, ZONES[7], trucks[2], RULES[5], parts[], floats[], pulses[], quips[], dir{gx,gy,tx,ty,targetZone,fixing,step,face}, selSensors, selRules, hintedOnce{}`.
`resetGame()` возвращает всё к исходному — рестарт без перезагрузки страницы.

### 6.4. Сервисный хук стенда
```js
window.__RR = {
  skipToSupply(), skipToRules(), skipToEnd(),   // быстрый перевод фаз для демонстраций
  state()  // снимок: mode, t, money, accidents, prevented, autoActs,
           // stock, trips, powerOut, активные rules, все зоны {id,on,v,failed,sensor}
}
```
Назначение: демо на выставке, автотесты, отладка балансировки.

### 6.5. Производительность и устойчивость
- DPR cap = 2; canvas в физических пикселях, логика — в CSS-пикселях.
- Cap частиц 800; троттлинг звука монет; экономия кадров в оверлеях.
- Нет аллокаций текстур/изображений — весь арт процедурный, мгновенный холодный старт.
- Автозавершение всех пауз таймером — игра не блокируется бездействием (kiosk-safe).

## 7. Метрики и телеметрия смены

| Метрика | Источник | Роль |
|---|---|---|
| p1, p2, p3 — % потерь по актам | `stat[n].lost/(lost+earned)` | Главное доказательство ценности лестницы |
| accidents | failZone/срывы | Тяжесть последствий |
| prevented | Превентивные действия | Качество ранней реакции |
| autoActs | Срабатывания правил | Вклад автоматики |
| tripsOnTime/tripsLate | Логистика | Клиентский показатель |
| money | Касса | Итоговый счёт |

Критерий «победы»: `p3 < p1` → «Логика окупилась». Отдельного game over нет — смена всегда доигрывается до конца (касса может уйти в минус: это тоже урок).

## 8. Известные ограничения прототипа и план развития

**Ограничения v1.0:**
- Баланс: при пассивном игроке автовыбор датчиков/правил делает Акт 3 почти самоиграющимся — приемлемо для стенда, слабо для соревновательного режима.
- Нет сохранения результатов/лидерборда, нет локализаций (RU only), нет настроек звука.
- Доступность: управление только курсором/клавиатурой, нет screen-reader-слоя (canvas).
- `stat.lost` смешивает штрафы и упущенную выгоду — на дебрифе стоит проговаривать состав «потерь».

**Roadmap (предложение):**
1. **Акт 4 «Оптимизация»** — тюнинг порогов правил игроком (обучение настройке автоматики, PDCA-цикл).
2. Пресеты сложности и seed-рандомизация отказов для реиграбельности.
3. Экспорт итогов (QR/ссылка) для сбора лидов со стенда; лидерборд тренинг-группы.
4. Локализация EN; вынос текстов в словарь.
5. Режим «пара»: директор + диспетчер на двух устройствах (разделение наблюдаемости и действия).
6. Встраиваемый дебриф-экран: авто-вопросы по фактическим ошибкам сессии (например, если игрок не выбрал датчик склада и потерял партию).
