# 1. Назначение продукта и учебная гипотеза

«Ручной режим» — браузерная обучающая аркада для руководителей производства, мастеров, инженеров, операционных менеджеров и команд улучшений. Завод представлен как система взаимозависимых потоков: оборудование, сырьё, люди, энергия, логистика, качество и заказы. Игрок проживает ключевой вывод: локальная героическая реакция не масштабируется, устойчивый результат появляется после изменения системы.

| Учебная гипотеза | Игрок лучше понимает Lean, когда сначала проживает последствия потерь и локальной оптимизации, а затем сам строит наблюдаемость, поток, вытягивание, стандарты и контуры решения проблем. |
|---|---|

## 1.1. Результаты обучения

- видеть производство как единый поток ценности, а не набор участков;
- различать симптом, отклонение, первопричину и контрмеру;
- понимать вред локальных KPI, лишнего WIP и постоянного героизма;
- применять визуальное управление, стандартную работу, pull, kanban, jidoka, heijunka, 5 Why, A3 и PDCA;
- связывать решения с lead time, WIP, OEE, FPY, OTIF, takt time и стоимостью потерь.

## 1.2. Продуктовые цели

| Цель | Измеримый признак |
|---|---|
| Обучение через действие | Не менее 70% игроков правильно выбирают системную контрмеру после главы. |
| Вовлечённость | Средняя сессия 20–35 минут; завершение первой главы выше 60%. |
| Практический перенос | Игрок формирует мини-A3 или план наблюдения для своего процесса. |
| Повторное прохождение | Сравниваются ручной, наблюдаемый и автоматизированный режимы. |
| Расширяемость | Новый учебный модуль добавляется без изменения ядра. |

# 2. Целевая аудитория и форматы

| Сегмент | Потребность | Режим |
|---|---|---|
| Начинающий руководитель | Понять, почему тушение пожаров не является управлением. | Сюжет + подсказки |
| Мастер / начальник смены | Отработать приоритизацию отклонений. | Сценарии смены |
| Lean / CI специалист | Показать командам эффекты flow, pull и стандартов. | Фасилитируемый тренинг |
| Студенты | Практика без доступа к реальному заводу. | Кампания + тесты |
| Топ-менеджмент | Связать системные решения с деньгами и сроками. | Executive-сценарии |

- Самостоятельная кампания на 8–12 часов.
- Корпоративный тренинг на 60–120 минут.
- Песочница с настройкой спроса, вариативности и ресурсов.
- Экзамен без подсказок и с отчётом по компетенциям.

# 3. Базовые принципы игрового дизайна

| Принцип | Реализация |
|---|---|
| Сначала проблема, потом термин | Игрок сначала переживает эффект; термин появляется в разборе. |
| Система важнее кликов | Скорость помогает только в первом акте. |
| Автоматизация требует стабильности | Нельзя автоматизировать непонятный и нестабильный процесс. |
| Проблемы становятся видимыми | Наблюдаемость снижает туман, но не исправляет процесс. |
| Локальный KPI может вредить | Максимальная загрузка станка может увеличить WIP и lead time. |
| Рефлексия обязательна | После уровня сравниваются ожидания, факты и контрмеры. |
| Ошибка безопасна | Провал объясняет механизм, а не только отнимает очки. |

| Драматургия | Хаос → видимость → стабильность → поток → вытягивание → качество в процессе → первопричины → непрерывное улучшение. |
|---|---|

# 6. Практическое обучение и перенос

| Фаза | Действие | Результат |
|---|---|---|
| Прогноз | Игрок выбирает ожидаемый исход. | Фиксация исходной модели мышления. |
| Эксперимент | Управляет симуляцией 3–8 минут. | Наблюдает причинно-следственную связь. |
| Разбор | Видит timeline и потери. | Понимает механизм. |
| Теория | Краткая карточка принципа. | Получает термин и контекст. |
| Перенос | Отвечает о своём процессе. | Связывает игру с работой. |
| Повтор | Запускает сценарий с контрмерой. | Закрепляет результат. |

| Шаблон практики | Отклонение → факты → влияние на клиента → причина → контрмера → проверка → новый стандарт. |
|---|---|

- сравнение ручного и системного режима;
- три крупнейшие потери;
- карта причин повторяющейся проблемы;
- контрмера и показатель проверки;
- план «что посмотреть на гемба завтра».

# 7. Игровые системы и экономика

| Система | Описание | Учебная функция |
|---|---|---|
| Деньги | Доход за выпуск и рейсы; потери от простоев, брака и срочности. | Связь процесса с бизнесом. |
| Поток заказа | Сырьё → обработка → сборка → склад → отгрузка. | Видимый lead time. |
| WIP | Единицы между процессами. | Очереди и скрытые проблемы. |
| Оборудование | Температура, износ, микростопы, аварии. | Стабильность и jidoka. |
| Люди | Нагрузка, усталость, навык. | Muri и развитие людей. |
| Качество | Дефект, обнаружение, переделка. | Цена позднего обнаружения. |
| Спрос | Ритм, mix, вариативность. | Takt, pull, heijunka. |
| Внимание | Ограниченный ресурс ручных действий. | Предел ручного контроля. |
| Доверие | Зависит от реакции руководителя. | Культура и видимость проблем. |

- Lead time = обработка + ожидание + транспорт + переделка.
- Throughput ограничивается эффективной мощностью ограничения.
- WIP растёт, когда upstream быстрее downstream.
- Риск дефекта растёт при перегрузке, нестабильности и износе.
- Стоимость проблемы увеличивается на последующих стадиях.

| Этап | Длительность | Состояние | Вывод |
|---|---|---|---|

| Акт 3 | ≈60 с | Логика работает, игрок ловит исключения. | Руководитель проектирует систему. |

| Зона | Сигнал | Проблема | Автоматизация |
|---|---|---|---|

| Бункер сырья | Уровень | Голодание линии | остаток≤20% → автозаказ |
| Погрузочный док | Дедлайн | Срыв OTIF | <5 с → автодиспетчер |
| Пресс «Циклоп» | Износ | Авария / дефект | TPM в следующих главах |
| Пост сборки | Силы | Перегрузка / ошибка | Балансировка |
| Склад-холод | Температура | Порча партии | T≥70% → усилить холод |

# 14. Балансировка и сложность

| Режим | Особенности |
|---|---|
| Учебный | Пауза времени, явные пороги, подсказки до ошибки. |
| Стандартный | Время продолжается, умеренная вариативность. |
| Эксперт | Шум датчиков, скрытые связи, ограниченный бюджет. |
| Экзамен | Без подсказок, фиксированный seed, оценка объяснения. |

- Первые 30 секунд обучают одному действию.
- Новая зона появляется после понимания предыдущей.
- Сложность растёт через взаимосвязи, а не только скорость.
- Любая авария имеет предшествующий сигнал.
- Системное решение превосходит идеальную скорость кликов на длинном горизонте.
